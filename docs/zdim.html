Z Data in Motion

As Enterprises deploy more and more applications in the cloud, the demand for access to CURRENT DATA becomes ever more critical. Terms Like "Data Fabric" and "Data Mesh" have entered the vernacular of IT professionals to describe architectural approaches for addressing that need. Sidestepping the precise academic meanings and subtleties of these terms, it is safe to say that most data fabrics fall into one of two broad catageories

  • Stream everything off the mainframe as fast as possible, and use modern, open source integration technologies to create a digital integration hub.
  • Leave the data in situ, and provide efficient APIs to access from wherever you need it.

Candidly, the first pattern is the most common pattern being implemented today. Advocates of this approach use a wide range of arguments to justify the approach, including "CQRS Architecture", "Getting off the Mainframe", "Cloud Modernisation". As a mainframe advocate I often find myself arguing in favour of the second approach on the basis of efficiency in not moving the data, and not incurring all the associated costs of software, hardware, encryption, governance etc... for all the data copying.

Truth is that both approaches have their merits for different use cases, and the worked examples in this site show how to deploy solutions to either the Z Data in Motion model or the Z Data at REST model. We also publish a hybrid model which combines both models to provide a comprehensive Environment for Realtime Information Services.

This section covers the three primary "Z Data in Motion" options for IBM Z data sources.

CDC Replication is aimed at heterogenous data replication.

  • It supports all the regular relational databases ( Db2 Oracle, SQL Server etc... ) as sources and targets.
  • Additionally, it supports non-relational databases such as IMS and VSAM.
  • And it supports non-relational targets such as Apache Kafka (various distributions of Kafka).

CDC Capture Agents stream changes to CDC Apply agents over TCPIP sockets. Devops management is supported with both graphical and scripting interfaces.

Q Replication is primarily aimed at homogeneous DB2 Replication. (both Db2 z/OS and DB2 LUW). If your sources and targets are all Db2, then Q Replication offers some attractive benefits such as:

  • It supports in-line DDL Replication between Db2 z/OS sources and Db2 z/OS targets..
  • It supports publishing to MQ in open formats, enabling a pub/sub integration model with MQ consumers.
  • It is the basis for the GDPS Active-Active continuous availability solution.

Q Capture Agents stream changes to Q Apply agents over Websphere MQ messaging channels. Devops management is supported with both graphical and scripting interfaces.

Datagate is a lightning fast data replication protocol between a Db2 z/OS V12 source and a Db2 Software Container, such as Db2 Analytics Accelerator, or Datagate.

  • It supports a choice of conventional row-based Db2 targets, or columnar Db2 targets that are optimised for Analytics.
  • It supports acceleration of analytical SQL queries on Db2 z/OS by transparently routing them to a columnar Db2 Software Container.
  • It provides the ability to place your operational Db2 z/OS data in the cloud to support new cloud-native development projects.

Datagate is a much narrower use case that the more generalised CDC and Q Replication tools, because it only supports Db2 Software containers as targets. But if your requirements fit within that envelope it is the best replication option by far ( lower z/OS software consumption charges, redcued cpu cycles, ultra-low data latency, devops controls for changes, etc... ).
