Building a Data Fabric for Mainframe Data

  • Z Data in Motion
  • Z Data at REST
  • ERIS

The diagram below depicts two of the most popular approaches to building a data fabric that incorporates data from IBM Z systems of record. A common approach is to move and replicate large amounts of data to external Digital Integration Hubs whose purpose is curate data to serve new cloud-based application services. Another approach is to serve data and transactional APIs directly from IBM Z, and it is important to note that the Z-centric approach also provides for patterns of consuming data in motion to derive curated data.

Z Data in Motion

Whichever Data Fabric pattern an enterprise uses, there will be a need to surface Z Data in Motion, to enable business events that are processed by systems of records to be surfaced for realtime business process and analytics. This section discusses the primary solutions that IBM offers for capturing and consuming "Z Data in Motion" whichever Data Fabric architecture is preferred.

Cloudpak for Data.

  • Is served by Db2 integrated synchronisation for access to current Db2 data in transactions or analytical formats.
  • Supports multiple optional services spannng data governance and AI.
  • Is deployable on any Openshift platform: Cloud, Midrange, Linux for Z etc...

Cloudpak for Integration

  • Accepts feeds from non-DB2 data.
  • Supports curation of data for services.
  • Publishes and Manages APIs on curated data

Z Data Mesh.

  • All APIs from raw Z data, or curated data, managed inside a secure Z hardware platform (ErisONE).
  • Platform for Data Owners to manage how data is published for consumption.
