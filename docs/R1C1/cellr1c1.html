Z Data in Motion

In recent years there has been an explosive growth in cloud applications, often with mobile phone apps as clients, that demand access to current data from operational datatores. Whether you are ordering a taxi, a pizza or a bank balance, the customer has zero interest in the state of the data 5 minutes ago. The only kind of data that is acceptable for most apps is data that is bang up to date.

Writing as a mainframe advocate, my instinctive reaction is to build APIs on the mainframe to serve those cloud applications directly. There is plenty of great software available from IBM to build REST APIs on top of various mainframe resources including CICS transactions, IMS transactions, Websphere programs, DB2 queries, Db2 stored procedures, VSAM and IMS data and so on. The Z Data at REST section provides details of those patterns.

However, it is an undeniable fact that the pattern most commonly adopted by enterprises today is to capture changes from mainframe databases, and stream them to a midrange or cloud platform, where the data can be received and manipulated to act as a source for serving APIs from that platform.

Arguing the relative merits of the different approaches is set aside for the purposes of the Z Data in Motion and Z Data at REST worked examples. However, the relative merits of moving data or accessing it in situ are considered in the ERIS Technical Zone, where a hybrid approach is presented.

The articles in this section contain worked examples of how to use various IBM software solutions to satisfy the pattern of streaming mainframe data off the mainframe to a remote consumer
